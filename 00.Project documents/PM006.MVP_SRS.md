## 第 1 章　文件說明

### 1.1 目的

本文件為《戰國三代記》MVP 階段的軟體需求規格書（SRS），供開發團隊作為設計、開發、測試的共同依據。

### 1.2 範圍

MVP 包含以下三個 Phase：

| Phase | 內容 | 週次 |
|-------|------|------|
| Phase 1 | 核心系統（遊戲入口、回合推進、存讀檔） | Week 3–6 |
| Phase 2 | 領地管理（三層領地、設施、年貢、資源結算） | Week 7–9 |
| Phase 3 | 基礎軍事（兵力系統、野戰、圍城、佔領） | Week 10–13 |

| 術語 | 定義 |

| 大名 | 玩家操控的最高統治者 |

## 第 2 章　業務需求（BA 撰寫）

### 2.1 業務背景

玩家扮演戰國時代的大名，目標是透過政治治理與軍事征戰，擴大自身勢力，最終統一天下。

MVP 階段的核心業務目標：讓玩家能體驗到「戰國領主」的基本統治循環——**管理封地、徵召兵力、發動戰役**。

### 2.2 業務流程（Business Flow）

```
【每月回合循環】

開始回合
  │
  ▼
事件通知（隨機事件、設施完工、軍事回報）
  │
  ▼
政務階段（玩家選擇 1~N 項政務指令）
  ├─ 治理郡城（升級設施、調整年貢）
  ├─ 家臣管理（指派知行、賞賜）
  └─ 召集農民兵（選擇徵召郡）
  │
  ▼
軍事階段（玩家選擇是否發動軍事行動）
  ├─ 出征（指定目標、家臣團、集合點）
  ├─ 野戰（攻郡）→ 戰鬥結算
  └─ 圍城（攻城）→ 每回合消耗守方資源
  │
  ▼
回合結算（系統自動執行）
  ├─ 資源結算（金錢、軍糧、人口）
  ├─ 農兵集結完成通知
  └─ 年貢收取
  │
  ▼
結束回合 → 月份 +1
```

### 3.2 主要 Use Case 詳述

---

#### UC-001：開始新遊戲

**前置條件**：玩家在主選單  
**主要流程**：
1. 玩家選擇「從頭開始」
2. 系統顯示可選劇本列表（含年份、說明）
3. 玩家選擇劇本
4. 系統顯示該劇本可選勢力列表（含大名名稱、領地數、初始兵力）
5. 玩家選擇勢力
6. 系統載入對應初始資料，進入遊戲主畫面
7. 系統顯示第一回合事件通知

**替代流程**：  
3a. 玩家返回主選單 → 結束 UC

**後置條件**：遊戲以所選勢力的初始狀態開始運行

---

#### UC-004：結束回合

**前置條件**：玩家在遊戲主畫面  
**主要流程**：
1. 玩家點擊「結束回合」
2. 系統執行回合結算序列：
   a. 資源結算（各郡產出 − 消耗 = 淨收入）
   b. 設施建設進度推進（在建設施 -1 月，完工則更新狀態）
   c. 農兵集結進度推進（集結中農兵 -1 月，完成則可出兵）
   d. 軍事行動推進（移動中部隊位置更新）
   e. 圍城消耗計算（守方軍糧 −圍城消耗，若歸零則觸發投降）
3. 系統更新月份（+1），並觸發當月事件
4. 系統顯示回合結算摘要面板
5. 玩家確認後進入下一回合

**例外流程**：  
2e. 若守方軍糧歸零 → 系統觸發「城池陷落」事件 → 轉至 UC-024（戰報）

---

#### UC-011：建設/升級設施

**前置條件**：玩家在郡詳情畫面，郡屬於己方，玩家金錢足夠  
**主要流程**：
1. 玩家在郡詳情畫面選擇「建設設施」
2. 系統顯示可建/可升的設施選項（含費用、效果、升級時間）
3. 玩家選擇目標設施
4. 系統顯示確認對話框（顯示扣除費用後的剩餘金錢）
5. 玩家確認
6. 系統扣除金錢，設施狀態改為「建設中（N 月）」
7. 玩家看到郡詳情更新

**替代流程**：  
1a. 郡已滿 5 個設施 → 系統顯示「無法再新增設施」  
3a. 金錢不足 → 選項呈現灰色，無法選取  
5a. 玩家取消 → 不扣費，回到步驟 2

**後置條件**：設施進入建設狀態，下 N 個月後完工並提升產出

---

#### UC-022：發動攻郡野戰

**前置條件**：玩家選定目標郡（敵方或中立），有可出兵的家臣團  
**主要流程**：
1. 玩家在地圖上選擇敵方郡
2. 系統顯示「攻郡」選項
3. 玩家選擇出兵家臣團（一個或多個一級家臣團）
4. 玩家指定集合地點（己方鄰接郡）
5. 玩家確認出兵命令
6. 系統登記軍事行動（下回合開始執行移動）
7. 部隊抵達目標郡時，系統自動觸發野戰結算：
   a. 計算攻守雙方有效兵力（基礎兵力 × 武將統率係數）
   b. 每回合攻守雙方互相造成損失
   c. 重複直到任一方兵力歸零或選擇撤退
8. 攻方勝利 → 領地所屬轉移，系統顯示戰報
9. 攻方失敗 → 殘存兵力撤回出發地，系統顯示戰報

**替代流程**：  
7c. 攻方選擇撤退 → 殘存兵力撤回，結束 UC

## 第 4 章　系統功能規格（SA 撰寫）

### 4.1 核心系統（CoreSystem）

#### 4.1.1 遊戲狀態機

```
[主選單] ──新遊戲/讀檔──▶ [遊戲中]
                              │
                   ┌──────────┼──────────┐
                   ▼          ▼          ▼
              [政務模式]  [地圖模式]  [戰鬥模式]
                   │                     │
                   └──────── 結束回合 ────┘
                              │
                         [回合結算]
                              │
                         [遊戲中] ← 下一回合
```

#### 4.1.2 回合結算序列（強制順序）

| 步驟 | 動作 | 說明 |
|------|------|------|
| 1 | 建設進度更新 | 在建設施每月 -1，完工時更新設施等級 |
| 2 | 農兵集結更新 | 集結中農兵每月 -1，完成後可出兵 |
| 3 | 軍隊移動更新 | 移動中部隊位置推進 |
| 4 | 圍城消耗計算 | 守方每月軍糧扣除圍城損耗值 |
| 5 | 資源產出結算 | 各郡依設施等級計算當月產出 |
| 6 | 資源消耗結算 | 扣除城兵維持費、軍事行動消耗 |
| 7 | 月份推進 | 月份 +1，12 月後歸 1 並年份 +1 |
| 8 | 事件觸發檢查 | 依當月觸發條件檢查事件 |

#### 4.1.3 存檔系統規格

| 項目 | 規格 |
|------|------|
| 存檔槽數量 | 手動 10 槽 + 自動存檔 1 槽 + 快速存檔 1 槽 |
| 存檔格式 | JSON，含 `schema_version` 欄位 |
| 存檔路徑 | `user://saves/slot_XX.json` |
| 存檔觸發 | 手動（玩家操作）、自動（每月回合結算後） |
| 存檔大小上限 | 10 MB |
| 讀取驗證 | 讀取時比對 `schema_version`，過舊版本執行 Migration |

### 4.2 領地系統（TerritoryModule）

#### 4.2.1 領地層級關係

```
令制國（Province）
├── 城（Castle）
│   ├── 郡 A（District）
│   ├── 郡 B
│   └── 郡 C
```

#### 4.2.2 郡資源產出規格

| 郡類型 | 主要產出 | 次要產出 | 設施類型 |

| 軍事郡 | 軍糧 | 城兵上限 | 軍事設施（駐屯地/兵營/軍陣營） |

|---------|------|--------|---------|---------|

| 商業設施 | 等級 | 升級費 | 升級時間 | 產出加成 |
|---------|------|--------|---------|---------|
| 市集 | Lv1 | — | — | 基礎值 |
| 商街 | Lv2 | 120 金 | 1 月 | +60% |
| 行會 | Lv3 | 360 金 | 2 月 | +150% |

| 軍事設施 | 等級 | 升級費 | 升級時間 | 城兵上限加成 |
|---------|------|--------|---------|------------|
| 駐屯地 | Lv1 | — | — | 基礎值 |
| 兵營 | Lv2 | 150 金 | 1 月 | +500 兵 |
| 軍陣營 | Lv3 | 400 金 | 2 月 | +1000 兵 |

> **限制**：每個郡最多同時擁有 5 個設施槽，且同類設施只能有 1 個。

#### 4.2.4 年貢系統規格

| 規則 | 說明 |
|------|------|
| 年貢率範圍 | 10% ～ 60%（玩家可自由設定） |
| 收取時機 | 每月回合結算時自動執行 |
| 忠誠度影響 | 年貢率 > 30% 時開始產生忠誠度下降壓力（MVP 簡化版） |
| 忠誠度公式（MVP 簡化） | `忠誠度變化 = -(年貢率 - 30%) × 0.5`（每月） |
| 叛亂觸發 | MVP 階段不實作自動叛亂（列入 Phase 4） |

### 4.3 軍事系統（MilitaryModule）

#### 4.3.1 兵力構成規格

| 兵種 | 來源 | 集結時間 | 軍糧負擔 | 上限計算 |
|------|------|---------|---------|---------|
| 城兵 | 城的常備軍 | 即時可用 | 勢力負擔 | 城等級 × 500 + 軍事設施加成 |
| 農民兵 | 郡的動員民兵 | 需 1 個月 | 前 3 月農民自負 | 郡人口 × 10% |

#### 4.3.2 家臣團兵力一覽

| 階級 | 個人兵力 | 可指揮上限 | 出兵單位 |
|------|---------|-----------|---------|
| 大名 | 5,000 | — | 本家家臣團 |
| 宿老 | 4,000 | 屬臣×4 + 下級×4 | 一級家臣團 |
| 家老 | 2,000 | 屬臣×3 + 下級×4 | 一級家臣團 |
| 部將 | 1,000 | 組頭×4 | 二級家臣團 |
| 侍大將 | 600 | 組頭×3 | 二級家臣團 |
| 組頭 | 200 | — | 無 |
| 直臣 | 1,000 | — | 本家直屬 |

> **出兵規則**：出兵單位為「家臣團」，不可單獨派遣個人。

#### 4.3.3 野戰結算規格

```
【野戰結算公式（MVP 版）】

有效攻擊力 = 攻方兵力 × (武將統率 / 100) × 地形係數
有效防禦力 = 守方兵力 × (武將統率 / 100) × 地形係數

每回合損失：
  守方損失 = 有效攻擊力 × 0.05  （±20% 隨機浮動）
  攻方損失 = 有效防禦力 × 0.05  （±20% 隨機浮動）

地形係數（MVP 版，簡化）：
  平地：1.0
  山地：守方 × 1.3（攻方佔劣勢）
  河川：攻方 × 0.8

結束條件：
  任一方兵力 ≤ 0 → 對方勝利
  攻方選擇撤退 → 守方勝利，攻方保留殘餘兵力
```

#### 4.3.4 圍城戰規格

```
【圍城戰機制（MVP 版）】

觸發條件：攻方選擇「圍城」而非「野戰」

每月消耗：
  守方軍糧消耗 = 守城兵力 × 0.1 + 人口 × 0.01

投降條件：
  守方軍糧 ≤ 0 → 守方投降，城池轉移給攻方

解圍條件：
  攻方軍糧耗盡 / 攻方主動撤軍 → 圍城解除

注意：攻方圍城期間，自身軍糧每月也會消耗（兵力 × 0.08）
```

#### 4.3.5 戰鬥結果處理

| 結果 | 攻方 | 守方 |
|------|------|------|
| 攻方勝利（野戰） | 佔領目標郡，更新領地所屬 | 殘存兵力撤至鄰接己方郡 |
| 攻方失敗（野戰） | 殘存兵力撤回出發郡 | 維持現狀 |
| 城池陷落（圍城） | 取得城及其下轄所有郡 | 守方武將可選擇投降或逃亡 |
| 圍城解除 | 撤回出發地 | 維持現狀，軍糧維持現有值 |

### 4.4 介面功能規格

## 第 6 章　系統架構（Architect 撰寫）

### 6.1 架構設計決策（ADR）

#### ADR-001：採用三層架構（Data / Logic / Presentation）

**決策**：嚴格分離資料層、邏輯層、展示層。

**理由**：
- 確保業務邏輯不被 UI 污染，未來可獨立替換 UI
- 利於單元測試：邏輯層可在無 UI 的情況下被測試
- 符合 Godot 4.x 的 Resource + Node 設計慣例

**後果**：
- Resource 類別（.gd）只存放資料，不含業務邏輯
- 所有計算集中在對應 Module
- UI 只透過 Signal 取得資料更新通知，不直接讀取 Module 內部狀態

---

#### ADR-002：採用命令模式（Command Pattern）處理玩家輸入

**決策**：所有玩家指令封裝為 `GameCommand` 物件，統一由 `CoreSystem` 派發。

**理由**：
- 統一入口，便於指令驗證（資源是否足夠、操作是否合法）
- 日後可實作「撤銷」功能或「指令重播」（Debug 用途）
- 便於記錄操作日誌

---

#### ADR-003：模組間不直接互相呼叫

**決策**：模組之間的溝通透過 `CoreSystem` 協調，不允許模組 A 直接呼叫模組 B 的方法。

**理由**：
- 避免循環依賴
- 降低模組耦合度，便於獨立開發與測試
- 未來新增模組時不需修改現有模組

**溝通方式**：
- 模組發出 `GameEvent`，由 `CoreSystem` 傳遞給其他相關模組
- UI 透過監聽 Signal 更新顯示

### 7.2 核心計算函數規格

#### 7.2.1 年貢計算

```gdscript
# TerritoryModule
func calculate_nengu(district: Territory) -> int:
    var base_output = _get_base_output(district)
    return int(base_output * district.nengu_rate)

func _get_base_output(district: Territory) -> int:
    var base = _BASE_OUTPUT[district.district_type]
    var multiplier = _get_facility_multiplier(district)
    return int(base * multiplier)

# 設施加成乘數
func _get_facility_multiplier(district: Territory) -> float:
    var multiplier = 1.0
    for facility in district.facilities:
        if not facility.is_constructing:
            multiplier += _FACILITY_BONUS[facility.type][facility.level]
    return multiplier
```

#### 7.2.2 野戰結算

```gdscript
# MilitaryModule
func resolve_field_battle(
    attacker: RetainerGroup,
    defender: RetainerGroup,
    terrain: TerrainType
) -> BattleResult:
    var att_power = attacker.forces * (attacker.leader.command / 100.0) * _terrain_factor(terrain, false)
    var def_power = defender.forces * (defender.leader.command / 100.0) * _terrain_factor(terrain, true)

    while attacker.forces > 0 and defender.forces > 0:
        var att_loss = int(def_power * 0.05 * randf_range(0.8, 1.2))
        var def_loss = int(att_power * 0.05 * randf_range(0.8, 1.2))
        attacker.forces -= att_loss
        defender.forces -= def_loss

    return BattleResult.new(attacker.forces, defender.forces)
```

### 7.3 邊界條件與錯誤處理規格

| 情境 | 處理方式 |
|------|---------|
| 升級設施時金錢不足 | CommandResult.success = false，訊息：「金錢不足」，不扣費 |
| 郡已有 5 個設施，嘗試再建 | CommandResult.success = false，訊息：「設施已達上限」 |
| 出兵時家臣團兵力為 0 | CommandResult.success = false，訊息：「無可出兵之兵力」 |
| 年貢率設定超過 60% | 強制截斷為 60%，顯示提示 |
| 存檔時磁碟空間不足 | 顯示錯誤訊息，不覆蓋原有存檔 |
| 讀取到不支援的 schema_version | 嘗試 Migration，若失敗顯示「存檔版本不相容」 |
| 兵力計算結果為負數 | 強制設為 0，視為全滅 |
| 圍城中攻方軍糧歸零 | 自動觸發「攻方撤退」，圍城解除 |
