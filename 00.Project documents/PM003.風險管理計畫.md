## 1. 風險管理框架

### 1.1 風險評估矩陣

| 衝擊程度 ↓ / 可能性 → | 低（<25%） | 中（25-60%） | 高（>60%） |
|----------------------|-----------|-------------|-----------|
| **高（阻塞開發）** | 中風險 🟡 | 高風險 🔴 | 極高風險 🔴🔴 |
| **中（影響時程）** | 低風險 🟢 | 中風險 🟡 | 高風險 🔴 |
| **低（輕微影響）** | 可接受 ⚪ | 低風險 🟢 | 中風險 🟡 |

### 1.2 風險應對策略

| 策略 | 說明 | 適用情境 |
|------|------|----------|
| **迴避** | 消除風險根源，調整計畫 | 極高風險，可規避 |
| **轉移** | 將風險責任轉給外部 | 法律、合約類風險 |
| **減緩** | 降低可能性或衝擊 | 大多數技術與進度風險 |
| **接受** | 知悉並監控，不主動處理 | 低風險或無法控制者 |

### 1.3 風險優先序計算

> 風險指數 = 可能性（1-5）× 衝擊（1-5）
> 
> - 1-5：低風險（監控）
> - 6-12：中風險（制定應對方案）
> - 13-25：高風險（立即採取行動）

## 2. 技術風險 (Technical Risks)

### T-001：家臣團系統邏輯複雜度超出預期
| 項目 | 內容 |
|------|------|
| **風險描述** | 家臣團的指揮鏈、忠誠度計算、派系互動涉及大量互相影響的變數，可能導致邏輯錯誤難以追蹤 |
| **可能性** | 高（4/5） |
| **衝擊** | 高（4/5） |
| **風險指數** | 16 🔴 |
| **觸發條件** | 忠誠度計算結果與預期不符、或派系事件循環觸發 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 為每個計算函數編寫單元測試<br>2. 建立「忠誠度計算追蹤日誌」（Debug 模式）<br>3. 將派系邏輯設計成獨立的狀態機，避免互相耦合<br>4. Phase 4 開發前召開設計審查 |
| **負責人** | Lead Developer |
| **監控頻率** | 每 Sprint |

### T-002：Godot 4.x 跨平台相容性問題
| 項目 | 內容 |
|------|------|
| **風險描述** | 在 Windows 上正常運行的遊戲，在 macOS 或 Linux 上可能出現路徑、字型、輸入法問題 |
| **可能性** | 中（3/5） |
| **衝擊** | 中（3/5） |
| **風險指數** | 9 🟡 |
| **觸發條件** | 在非 Windows 環境測試時出現 UI 錯位、中文顯示異常 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 每個 Phase 結束後在三平台各測試一次<br>2. 使用 Godot 內建的跨平台路徑 API，避免硬編碼路徑分隔符<br>3. 字型採用支援繁體中文的免費字型（如思源宋體）並打包進遊戲資源 |
| **負責人** | Lead Developer |
| **監控頻率** | 每 Phase 結束 |

### T-003：效能問題（大型勢力回合計算超時）
| 項目 | 內容 |
|------|------|
| **風險描述** | 當遊戲進入中後期，50 個勢力、各有 500 名家臣，同時計算忠誠度 + 派系 + AI 決策，可能超過 3 秒目標 |
| **可能性** | 中（3/5） |
| **衝擊** | 高（4/5） |
| **風險指數** | 12 🟡 |
| **觸發條件** | 回合計算時間在 Debug 模式下超過 2 秒 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 從 Phase 3 起在每個 Sprint 結束後進行效能分析<br>2. 使用 Godot 的 Profiler 找出熱點函數<br>3. 設計「延遲計算」機制，非玩家勢力只在必要時計算<br>4. 大型陣列計算考慮分幀處理（`await get_tree().process_frame`）|
| **負責人** | Lead Developer |
| **監控頻率** | 每 Sprint |

### T-004：存檔格式不穩定（版本升級導致存檔不相容）
| 項目 | 內容 |
|------|------|
| **風險描述** | 開發過程中資料結構改變，導致舊存檔在新版本無法讀取，玩家流失 |
| **可能性** | 高（4/5） |
| **衝擊** | 中（3/5） |
| **風險指數** | 12 🟡 |
| **觸發條件** | 任何 Resource 類別的欄位名稱或型別變更 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 存檔格式加入 `schema_version` 欄位<br>2. 建立 Migration 機制（舊格式 → 新格式的轉換函數）<br>3. 任何資料結構變更必須更新版本號並寫 Migration<br>4. Beta 前鎖定存檔格式（不再做破壞性變更）|
| **負責人** | Lead Developer |
| **監控頻率** | 每次資料結構修改時 |

## 3. 時程風險 (Schedule Risks)

### S-001：需求蔓延（Scope Creep）
| 項目 | 內容 |
|------|------|
| **風險描述** | 開發過程中不斷加入新功能點子，導致時程失控 |
| **可能性** | 高（5/5） |
| **衝擊** | 高（4/5） |
| **風險指數** | 20 🔴 |
| **觸發條件** | 單一 Phase 新增功能超過原計畫 10% |
| **應對策略** | 迴避 + 減緩 |
| **具體措施** | 1. 所有功能變更必須走 Change Request 流程，由 PM 審核<br>2. 未入本階段 Backlog 的功能一律列入 P2 / 下一版本<br>3. 每週 Sprint Review 時明確展示「本 Sprint 完成 vs 計畫」<br>4. PM 有權凍結需求（在 M3 之後不接受核心功能新增）|
| **負責人** | PM (AustinLiao69) |
| **監控頻率** | 每週 |

### S-002：技術卡關導致進度落後
| 項目 | 內容 |
|------|------|
| **風險描述** | 遇到技術難題（如 Godot 特定 Bug、GDScript 限制），花費超過預期時間 |
| **可能性** | 中（3/5） |
| **衝擊** | 中（3/5） |
| **風險指數** | 9 🟡 |
| **觸發條件** | 單一任務超時超過 2 倍預估時間 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 每個 Phase 預留 20% 緩衝時間<br>2. 技術卡關超過 1 天，立即向 PM 報告並討論替代方案<br>3. 建立「技術問題追蹤清單」，每日更新狀態<br>4. 難度較高的功能優先排入 Sprint 前半段 |
| **負責人** | Lead Developer |
| **監控頻率** | 每日 |

### S-003：設計變更衝擊開發中的系統
| 項目 | 內容 |
|------|------|
| **風險描述** | 設計凍結後仍對已開發的系統進行規格修改，導致重工 |
| **可能性** | 中（3/5） |
| **衝擊** | 高（4/5） |
| **風險指數** | 12 🟡 |
| **觸發條件** | 在 M0 設計凍結後，對已進入開發的功能提出規格修改 |
| **應對策略** | 減緩 |
| **具體措施** | 1. M0 後任何設計修改需填寫「設計變更申請（DCR）」<br>2. PM 評估影響範圍後才決定是否接受<br>3. 已接受的 DCR 進入下一 Sprint，不插入當前 Sprint<br>4. 重大設計變更（影響超過 1 個模組）需召開設計審查 |
| **負責人** | PM (AustinLiao69) |
| **監控頻率** | 每 Sprint |

## 4. 品質風險 (Quality Risks)

### Q-001：遊戲平衡性問題（數值過強或過弱）
| 項目 | 內容 |
|------|------|
| **風險描述** | 忠誠度、年貢率、軍力等數值設計失衡，導致遊戲體驗單調或挫敗感過強 |
| **可能性** | 高（4/5） |
| **衝擊** | 中（3/5） |
| **風險指數** | 12 🟡 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 所有數值參數以設定檔（.tres / JSON）管理，不寫死在程式碼中<br>2. 建立「平衡性試算表」，每個數值有公式來源與設計理由<br>3. Alpha 後安排 3 位以上的內部測試者進行平衡測試<br>4. Beta 建立玩家回饋收集機制（問卷 / Steam 論壇） |
| **負責人** | Game Designer |
| **監控頻率** | Alpha 後每週 |

### Q-002：關鍵 Bug 影響正式發布
| 項目 | 內容 |
|------|------|
| **風險描述** | 正式發布時存在阻塞性 Bug，導致負評或退款潮 |
| **可能性** | 中（3/5） |
| **衝擊** | 極高（5/5） |
| **風險指數** | 15 🔴 |
| **觸發條件** | Release Candidate 測試中發現 P0 Bug |
| **應對策略** | 減緩 |
| **具體措施** | 1. 建立 Bug 優先序分級（P0=阻塞發布 / P1=嚴重 / P2=一般）<br>2. Release Candidate 階段：P0 Bug 必須清零才能發布<br>3. Phase 7 整合優化週期不少於 3 週<br>4. 建立「發布前檢查清單」並逐項確認 |
| **負責人** | QA Lead |
| **監控頻率** | Beta 後每日 |

## 5. 市場風險 (Market Risks)

### M-001：玩家接受度低於預期
| 項目 | 內容 |
|------|------|
| **風險描述** | 遊戲的深度機制對大眾玩家過於複雜，導致好評率低、口碑不佳 |
| **可能性** | 中（3/5） |
| **衝擊** | 高（4/5） |
| **風險指數** | 12 🟡 |
| **應對策略** | 減緩 |
| **具體措施** | 1. MVP 完成後（M3）即進行首次玩家可玩性測試<br>2. 設計漸進式教學系統（Tutorial），降低上手門檻<br>3. 提供「簡易模式」選項，簡化部分機制<br>4. Alpha 前在目標社群（戰國歷史論壇、策略遊戲 Discord）尋求早期回饋 |
| **負責人** | PM (AustinLiao69) |
| **監控頻率** | Alpha 後 |

### M-002：競品搶先發布類似產品
| 項目 | 內容 |
|------|------|
| **風險描述** | 開發期間，市場上出現功能相近的競品，侵蝕目標市場 |
| **可能性** | 低（2/5） |
| **衝擊** | 中（3/5） |
| **風險指數** | 6 🟢 |
| **應對策略** | 接受（持續監控） |
| **具體措施** | 1. 每季進行競品監控（Steam 新遊戲追蹤）<br>2. 加強《戰國三代記》的差異化特色（家臣政治深度）<br>3. 若有重大競品出現，召開產品方向審查 |
| **負責人** | PM (AustinLiao69) |
| **監控頻率** | 每季 |

## 6. 資源風險 (Resource Risks)

### R-001：Solo Developer 精力耗盡（Burnout）
| 項目 | 內容 |
|------|------|
| **風險描述** | 在小型或獨立開發的情況下，長期高強度工作導致效率下滑或停止開發 |
| **可能性** | 高（4/5） |
| **衝擊** | 極高（5/5） |
| **風險指數** | 20 🔴 |
| **應對策略** | 減緩 |
| **具體措施** | 1. 嚴格遵守開發工作日時間，設定每週工時上限<br>2. 每個 Phase 結束後安排 2-3 天休息期<br>3. 將任務分解至每日可完成的粒度，維持完成感<br>4. 建立「最小可交付目標」，在精力有限時優先完成 P0 功能 |
| **負責人** | PM (AustinLiao69) |
| **監控頻率** | 每週自我評估 |

## 8. 設計變更申請（DCR）流程

```
1. 提出者填寫 DCR 表單（功能描述 / 影響範圍 / 緊急程度）
2. PM 在 2 個工作日內評估
3. 評估結果：接受 → 排入下 Sprint / 拒絕 → 進入 P2 Backlog
4. 接受的 DCR 更新設計文件版本號
5. 在下一 Sprint Review 中驗收
```
