## 1. 技術選型

| 項目 | 選擇 | 理由 |
|------|------|------|
| **遊戲引擎** | Godot 4.x | 開源免費、跨平台、GDScript 開發效率高 |
| **開發語言** | GDScript | 與 Godot 深度整合，適合遊戲邏輯 |
| **版本控制** | Git + GitHub/Replit | 標準業界工具，支援協作與版本回溯 |
| **資料格式** | Godot Resource (.tres) | 原生整合、型別安全、Godot 編輯器可視化編輯 |
| **存檔格式** | JSON + 版本號 | 人類可讀、跨版本 Migration 容易 |
| **字型** | 思源宋體 / Noto Serif TC | 支援繁體中文、授權友善 |

## 2. 系統架構總覽

### 2.1 三層架構

```
┌─────────────────────────────────────────┐
│          展示層 (Presentation Layer)      │
│  UIManager │ MapView │ PanelSystem       │
│  CharacterPanel │ BattleUI │ EventPopup  │
└────────────────────┬────────────────────┘
                     │ Signal / Method Call
┌────────────────────▼────────────────────┐
│          邏輯層 (Game Logic Layer)        │
│  CoreSystem (指令處理 / 回合推進)         │
│  ┌──────────┬──────────┬──────────┐     │
│  │Retainer  │Territory │Military  │     │
│  │Module    │Module    │Module    │     │
│  ├──────────┼──────────┼──────────┤     │
│  │Intrigue  │Council   │Intel     │     │
│  │Module    │Module    │Module    │     │
│  └──────────┴──────────┴──────────┘     │
└────────────────────┬────────────────────┘
                     │ Resource Load/Save
┌────────────────────▼────────────────────┐
│          資料層 (Data Layer)             │
│  GameData │ Faction │ Retainer          │
│  Territory │ Facility │ SaveSystem      │
└─────────────────────────────────────────┘
```

### 2.2 核心設計原則

1. **資料與邏輯分離**：Resource（資料）不含業務邏輯，邏輯由對應 Module 處理
2. **模組間低耦合**：模組透過 `CoreSystem` 溝通，不直接互相呼叫
3. **Signal 驅動 UI**：UI 監聽 Signal，不直接讀取 Module 內部狀態
4. **命令模式（Command Pattern）**：玩家操作封裝為 `GameCommand` 物件，便於記錄、撤銷

## 4. 核心資料結構

### 4.1 資料類層級

```gdscript
GameData (Resource)             # 整局遊戲資料的根節點
├── factions: Array[Faction]    # 所有勢力
├── current_date: GameDate      # 當前日期（年/月）
├── player_faction_id: String   # 玩家操控的勢力 ID
└── schema_version: int         # 存檔格式版本號

Faction (Resource)
├── id: String
├── name: String
├── daimyo_id: String           # 大名的 Retainer ID
├── gold: int
├── military_supply: int
├── territories: Array[String]  # Territory ID 陣列
└── retainers: Array[Retainer]

Retainer (Resource)
├── id: String
├── name: String
├── rank: RetainerRank (enum)   # 大名/宿老/家老/部將/組頭
├── lineage: Lineage (enum)     # 一門/譜代/外樣
├── loyalty: int                # 0-100
├── ambition: int               # 0-100（野心值）
├── faction_id: String          # 所屬派系
├── abilities: RetainerAbilities
│   ├── martial: int            # 武力
│   ├── command: int            # 統率
│   ├── intelligence: int       # 智謀
│   └── politics: int           # 政務
└── forces: RetainerForces
    ├── castle_forces: int      # 城兵（立即可用）
    └── peasant_forces: int     # 農兵（需集結）

Territory (Resource)
├── id: String
├── name: String
├── type: TerritoryType (enum)  # 令制國/城/郡
├── district_type: DistrictType # 農業郡/商業郡/軍事郡
├── level: int
├── assigned_retainer_id: String
├── nengu_rate: float           # 年貢率 0.0-1.0
├── facilities: Array[Facility]
└── parent_id: String           # 上層領地 ID
```

### 4.2 命令物件（Command Pattern）

```gdscript
class_name GameCommand extends Resource
enum CommandType {
    END_TURN,
    ASSIGN_RETAINER,
    UPGRADE_FACILITY,
    SET_NENGU_RATE,
    MOBILIZE_FORCES,
    LAUNCH_ATTACK,
    DISPATCH_NINJA,
    REWARD_RETAINER,
    PROMOTE_RETAINER,
    HOLD_COUNCIL,
}

var type: CommandType
var actor_id: String        # 發令者（通常是玩家勢力）
var target_id: String       # 命令對象（家臣/領地/敵方）
var parameters: Dictionary  # 命令參數（視命令類型而異）
var timestamp: GameDate     # 發令時間
```

## 5. 模組介面規格（Module Interface Spec）

每個模組都必須實作以下標準介面：

```gdscript
class_name BaseModule extends Node

# 初始化：傳入 GameData 的參考
func initialize(game_data: GameData) -> void: pass

# 處理命令：回傳是否成功
func handle_command(command: GameCommand) -> CommandResult: pass

# 回合結算：每月自動執行
func process_turn(game_data: GameData) -> Array[GameEvent]: pass

# 取得模組狀態（供 UI 讀取）
func get_state() -> Dictionary: pass
```

### 5.1 CommandResult 結構

```gdscript
class_name CommandResult extends Resource
var success: bool
var message: String             # 給玩家看的結果訊息
var events: Array[GameEvent]    # 此命令觸發的事件
var state_changes: Dictionary   # 狀態變更摘要（供 UI 更新）
```

### 5.2 GameEvent 結構

```gdscript
class_name GameEvent extends Resource
enum EventType {
    LOYALTY_CHANGE,
    FACTION_CONFLICT,
    RETAINER_REVOLT,
    BATTLE_RESULT,
    INTEL_REPORT,
    TERRITORY_CHANGE,
    RESOURCE_CHANGE,
    HISTORICAL_EVENT,
}

var type: EventType
var title: String
var description: String
var choices: Array[EventChoice]    # 玩家可選項（可為空）
var auto_resolve: bool             # true = 自動處理，false = 需玩家選擇
```

## 6. 資料流（Data Flow）

### 6.1 玩家命令流程

```
玩家操作 UI
    ↓
UIManager 建立 GameCommand 物件
    ↓
CoreSystem.execute_command(command)
    ↓
CoreSystem 驗證命令合法性
    ↓ （合法）
對應 Module.handle_command(command)
    ↓
Module 更新 GameData
Module 產生 Array[GameEvent]
    ↓
CoreSystem 處理 GameEvent
    ↓
emit_signal("game_state_changed", state_changes)
    ↓
UIManager 監聽 Signal，更新顯示
```

### 6.2 回合結算流程

```
玩家點擊「結束回合」
    ↓
CoreSystem.process_turn()
    ↓ 依序呼叫
TerritoryModule.process_turn()   → 資源結算、設施完工
RetainerModule.process_turn()    → 忠誠度更新、士兵補充
MilitaryModule.process_turn()    → 軍事行動推進
IntrigueModule.process_turn()    → 派系動態、叛變檢查
IntelModule.process_turn()       → 忍者任務結果
CouncilModule.process_turn()     → 評定議題觸發
    ↓
CoreSystem 收集所有 GameEvent
CoreSystem 推進日期（月份 +1）
    ↓
UIManager 顯示回合結算結果
```

## 7. 存檔系統設計

### 7.1 存檔格式

```json
{
  "schema_version": 1,
  "save_date": "2026-04-29T10:00:00",
  "game_data": {
    "current_date": { "year": 1560, "month": 5 },
    "player_faction_id": "oda",
    "factions": [ ... ],
    "territories": [ ... ]
  }
}
```

### 7.2 存檔路徑

```
user://saves/slot_01.json  ← 玩家手動存檔（10 槽）
user://saves/autosave.json ← 每月自動存檔（1 槽）
user://saves/quicksave.json← 快速存檔（1 槽）
```

### 7.3 Migration 機制

```gdscript
class_name SaveMigration

static func migrate(data: Dictionary) -> Dictionary:
    var version = data.get("schema_version", 0)
    if version < 1:
        data = _migrate_v0_to_v1(data)
    if version < 2:
        data = _migrate_v1_to_v2(data)
    return data

static func _migrate_v0_to_v1(data: Dictionary) -> Dictionary:
    # 轉換邏輯...
    data["schema_version"] = 1
    return data
```

### 8.1 命名規則

| 類型 | 規範 | 例子 |
|------|------|------|
| 類別名稱 | PascalCase | `RetainerModule`, `GameCommand` |
| 函數名稱 | snake_case | `calculate_loyalty()`, `get_total_forces()` |
| 變數名稱 | snake_case | `current_loyalty`, `max_forces` |

```
main (穩定版，只接受 merge)

```

  feat     新功能
  fix      修復 Bug
  refactor 重構（不影響功能）

  feat: 新增忠誠度 6 段系統

- 所有 Level 1 DoD 條件通過
- Code Review 完成（自我審查或同伴審查）
- 不破壞 main branch 的基本流程

## 10. 效能最佳化指引

### 10.1 資料計算最佳化

```gdscript
# ❌ 避免：每幀重複計算大型陣列
func _process(delta):
    var total = 0
    for retainer in all_retainers:
        total += retainer.calculate_forces()

# ✅ 建議：快取計算結果，只在資料變動時更新
var _cached_total_forces: int = -1

func get_total_forces() -> int:
    if _cached_total_forces < 0:
        _cached_total_forces = _calculate_total_forces()
    return _cached_total_forces

func _on_retainer_forces_changed():
    _cached_total_forces = -1  # 使快取失效
```

### 10.2 回合計算最佳化

- 非玩家勢力的忠誠度計算可採用「懶計算」（只在需要顯示時才計算）
- 大型陣列處理超過 1000 項時，考慮分幀執行（使用 `await`）
- Godot Profiler 在每個 Phase 結束時執行一次，確認熱點函數
