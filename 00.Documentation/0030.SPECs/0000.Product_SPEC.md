
# 《戰國三代記》Product SPEC v1.0

## 1. 產品概述 (Product Overview)

### 1.1 遊戲類型與目標市場
- **遊戲類型**：戰略模擬遊戲 (Strategy Simulation Game)
- **平台**：PC (Windows, macOS, Linux)
- **開發引擎**：Godot 4.x
- **開發語言**：GDScript
- **目標市場**：喜愛日本戰國歷史、策略遊戲和政治權謀的玩家

### 1.2 核心價值主張
- 深度的封建社會模擬，強調家臣團內部政治
- 多層級的領地管理系統 (令制國 → 城 → 郡)
- 豐富的權謀系統，包含忠誠度、派系鬥爭、叛變風險
- 真實的戰國時代軍事系統，區分城兵與農兵

### 1.3 競品分析與差異化定位
- 與《信長之野望》相比：更深入的家臣團內部政治
- 與《全軍破敵》相比：更專注於封建社會結構
- 與《十字軍之王》相比：更聚焦於日本戰國特色

### 2.3 核心設計原則

1. **資料與邏輯分離**：Resource（資料）不含業務邏輯，邏輯由對應 Module 處理
2. **模組間低耦合**：模組透過 `CoreSystem` 溝通，不直接互相呼叫
3. **Signal 驅動 UI**：UI 監聽 Signal，不直接讀取 Module 內部狀態
4. **命令模式（Command Pattern）**：玩家操作封裝為 `GameCommand` 物件，便於記錄與除錯

### 2.5 核心資料結構（GDScript Resource）

```gdscript

class_name GameData extends Resource

var military_supply: int
var territories: Array[String]      # Territory ID 陣列

# ── 家臣 ────────────────────────────────────────
class_name Retainer extends Resource
var id: String
var name: String
var rank: RetainerRank              # 見 SRS_0003
var lineage: LineageType            # 一門/譜代/外樣
var loyalty_value: int              # 0–100，內部數值，玩家不可見
var superior_id: String             # 直屬上司 ID
var subordinates: Array[String]

# ── 武將評價（四面向，玩家只見文字詞彙）──────────
class_name RetainerEval extends Resource
var military_score: int             # 軍事  0–100（內部，不顯示）
var strategy_score: int             # 智謀  0–100
var politics_score: int             # 政治  0–100
var charisma_score: int             # 魅力  0–100
# ↑ 對應 SRS_0011 詞彙表（E/D/C/B/A/S → 文字評價）

# ── 領地 ────────────────────────────────────────
class_name Territory extends Resource
var id: String
var name: String
var type: TerritoryType             # 令制國/城/郡
var district_type: DistrictType     # 農業/商業/軍事（郡層級有效）
var level: int                      # 城等級 Lv1–Lv3
var assigned_retainer_id: String
var nengu_rate: float               # 0.1–0.5
var facilities: Array[Facility]
var parent_id: String               # 上層領地 ID
```

### 2.6 命令物件（Command Pattern）

```gdscript
class_name GameCommand extends Resource

enum CommandType {
    END_TURN,
    ASSIGN_RETAINER,
    UPGRADE_FACILITY,
    SET_NENGU_RATE,
    MOBILIZE_FORCES,
    LAUNCH_ATTACK,
    DISPATCH_NINJA,
    REWARD_RETAINER,
    PROMOTE_RETAINER,
    HOLD_COUNCIL,
    POST_WAR_DECIDE,
}

var type: CommandType
var actor_id: String        # 發令者（通常是玩家勢力）
var target_id: String       # 命令對象（家臣/領地/敵方）
var parameters: Dictionary  # 命令參數（視命令類型而異）
var timestamp: Dictionary   # {year: int, month: int}

# ── 命令結果 ─────────────────────────────────────
class_name CommandResult extends Resource
var success: bool
var message: String
var events: Array[GameEvent]
var state_changes: Dictionary

# ── 遊戲事件 ─────────────────────────────────────
class_name GameEvent extends Resource
enum EventType {
    LOYALTY_CHANGE,
    FACTION_CONFLICT,
    RETAINER_REVOLT,
    BATTLE_RESULT,
    INTEL_REPORT,
    TERRITORY_CHANGE,
    RESOURCE_CHANGE,
    POST_WAR_DECISION,
    EVAL_EVOLVED,
}
var type: EventType
var title: String
var description: String
var choices: Array[EventChoice]
var auto_resolve: bool
```

### 2.7 資料流

#### 玩家命令流程
```
玩家操作 UI
    ↓
UIManager 建立 GameCommand 物件
    ↓
CoreSystem.execute_command(command)
    ↓
CoreSystem 驗證命令合法性
    ↓（合法）
對應 Module.handle_command(command)
    ↓
Module 更新 GameData → 產生 Array[GameEvent]
    ↓
CoreSystem 處理 GameEvent
    ↓
emit_signal("game_state_changed", state_changes)
    ↓
UIManager 監聽 Signal，更新顯示
```

#### 回合結算流程
```
玩家點擊「結束回合」
    ↓
CoreSystem.process_turn()
    ↓ 依序呼叫
TerritoryModule.process_turn()    → 資源結算、設施完工
RetainerModule.process_turn()     → 忠誠度更新、士兵補充
MilitaryModule.process_turn()     → 軍事行動推進
IntrigueModule.process_turn()     → 派系動態、叛變檢查
IntelModule.process_turn()        → 忍者任務結果
CouncilModule.process_turn()      → 評定議題觸發
FactionModule.process_turn()      → 派系勢力更新
RetainerEvalModule.process_turn() → 評價演化檢查
PostWarModule.process_turn()      → 統治衝擊恢復
    ↓
CoreSystem 收集所有 GameEvent，推進日期（月份 +1）
    ↓
UIManager 顯示回合結算結果
```


#### Migration 機制
```gdscript
class_name SaveMigration
static func migrate(data: Dictionary) -> Dictionary:
    var version = data.get("schema_version", 0)
    if version < 1:
        data = _migrate_v0_to_v1(data)
    return data
```

## 3. 功能需求規格 (Functional Requirements)

### 3.1 遊戲入口功能
- 新遊戲創建
- 存檔讀取 (最多10個存檔槽)

### 3.2 政務管理功能

#### 3.2.1 基本政務
| 功能 | 描述 | 輸入 | 輸出 |
|------|------|------|------|
| 任命官職 | 指派家臣擔任職位 | 家臣ID, 職位類型 | 家臣忠誠度變化 |
| 治理郡城 | 管理領地發展 | 領地ID, 治理指令 | 領地等級/資源變化 |
| 施行法令 | 頒布政策影響全勢力 | 法令類型, 參數 | 全體忠誠度/資源變化 |
| 視察地方 | 巡視領地觸發事件 | 領地ID | 隨機事件觸發 |

#### 3.2.2 高級政務
| 功能 | 描述 | 業務邏輯 |
|------|------|----------|
| 舉行典禮 | 提升威望和忠誠度 | 消耗金錢，提升全體忠誠度 |

### 3.3 軍事系統功能

#### 3.3.1 兵力管理
```gdscript
# 兵力類型定義
enum ForceType {
    CASTLE_FORCES,    # 城兵：立即可用，由勢力負擔軍糧
    PEASANT_FORCES    # 農兵：需1個月集結，前3月自負軍糧
}

# 兵力計算
func calculate_available_forces(territory: Territory, force_type: ForceType) -> int:
    match force_type:
        ForceType.CASTLE_FORCES:
            return territory.castle_level * 500
        ForceType.PEASANT_FORCES:
            return territory.population * 0.1
```

#### 3.3.2 戰鬥系統
| 戰鬥類型 | 觸發條件 | 介面類型 | 勝利條件 |
|---------|----------|----------|----------|
| 野戰 | 攻擊郡 | 戰棋式地圖 | 敵軍全滅或撤退 |
| 圍城戰 | 包圍城池 | 回合制消耗 | 敵方軍糧耗盡 |
| 攻城戰 | 強攻城池 | 分層攻防 | 攻破本丸 |

## 4. 用戶體驗設計 (UX Design Requirements)

### 4.1 介面設計原則
- **戰情室風格**：古風地圖攤在桌上，立體旗幟標示
- **分層導航**：令制國 → 城 → 郡的階層式瀏覽
- **資訊密度適中**：重要資訊一目了然，詳細資訊點擊查看

### 4.2 主要介面規劃

#### 4.2.1 主地圖介面
- 全國地圖模式：顯示勢力分布
- 地方地圖模式：顯示城郡詳情
- 軍勢標示：立體軍旗顯示部隊位置
- 切換按鈕：勢力圖/外交圖/分封圖

#### 4.2.2 政務介面
- 常駐命令列：存檔、讀檔、情報查看
- 政務選單：樹狀結構，最多三層
- 執行確認：重要決策需確認對話框

#### 4.2.3 戰鬥介面
- 野戰：六角格戰棋，地形效果明顯
- 攻城：分層城防圖，進攻路線清楚
- 部隊資訊：血量、士氣、武將能力一目了然

### 4.3 操作流程設計

#### 4.3.1 典型遊戲回合流程
```
1. 回合開始 → 事件提示 → 收入結算
2. 政務階段 → 選擇政務項目 → 執行決策
3. 軍事階段 → 部隊移動 → 戰鬥解決
4. 評定階段 → 家臣提案 → 大名決策
5. 情報階段 → 忍者報告 → 指派新任務
6. 回合結束 → 自動存檔 → 下一回合
```

#### 4.3.2 家臣管理流程
```
查看家臣列表 → 選擇家臣 → 查看詳細資訊
→ 執行操作（晉升/指派/獎賞） → 確認變更
→ 查看忠誠度變化 → 更新家臣團結構
```
